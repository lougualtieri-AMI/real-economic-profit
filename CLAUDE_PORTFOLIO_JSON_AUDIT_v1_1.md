# CLAUDE PORTFOLIO JSON AUDIT PROMPT — v1.1
**Purpose: Verify and integrate new extraction data into the master REPA portfolio JSON. Pure JSON in, JSON out. No PDFs, no Gemini prompt, no Report 3.**

> **Changelog from v1.0 (May 26, 2026):** Part 2 schema list (which enumerated field names matching neither the extraction output nor the app — e.g. `da_total`, `restructuring_total`, `gaap_net_income`, `period_end_price`) replaced with a pointer to the REPA Field Dictionary (v1.0). Part 3 framework rules reworded to reference the app's actual fields. Cross-references updated to the Claude Audit Prompt (v1.0), Historical Backtest (v2.1), and Annotation (v1.3).

---

## ⚠️ FRAMING — READ FIRST

This prompt does ONE thing: it audits new extraction data against the REPA framework rules and integrates verified years into the master portfolio JSON. It is the lightweight counterpart to the full audit cycle, used when:

- A new 10-K has been extracted (NotebookLM) and needs to be added to an existing portfolio entry
- An older year is being backfilled into the master
- A schema correction needs to be applied to one company's existing data
- You want to verify a candidate extraction against the master before committing

This prompt does NOT generate the three-PDF report suite, the Gemini Deep Research prompt, or the Annotated companion. For those, use the Claude Audit Prompt (v1.0) — ANNUAL mode for a full annual cycle, QUARTERLY mode for a TTM from a 10-Q.

---

## PART 0 — REQUIRED INPUTS

The audit cannot proceed without all three of:

1. **Master portfolio JSON** (current, post-backup) — the system of record
2. **Candidate extraction(s)** — one or more individual extraction JSONs from NotebookLM, OR a multi-year extraction file
3. **REPA Brief** for the company (PDF or MD) — generated by the REPA app after the candidate extractions were loaded; used as a reconciliation check

If any of these is missing, request it before proceeding. Do NOT fall back to producing isolated single-company JSON output. Standard request:

> "Portfolio JSON audit requires: (a) current master portfolio JSON, (b) candidate extraction(s), (c) REPA Brief reflecting the candidate data. Missing: [list]. Upload to proceed."

**Backup confirmation:** Before producing the updated master, confirm with the user that the current master has been backed up. One-line check, not a paragraph.


If the candidate covers multiple companies (rare), surface this immediately and confirm with the user before proceeding. Default assumption is single-company.

## PART 2 — SCHEMA VERIFICATION

Verify each candidate year against the **REPA Field Dictionary (v1.0)** — the canonical field set, which maps every extraction field name to its app field and notes whether it lands in `formData` or `extra`.

The candidate extraction uses the **NotebookLM Extraction Prompt's** field names: Title Case for an annual 10-K extraction ("Cash From Operations", "D&A", "Non-Current Marketable Securities") or snake_case for a TTM ("cash_from_operations", "d_and_a", …). The master stores the **app's** short field names (`cfo`, `da`, …). The app's import mapper bridges the two; this audit confirms the candidate carries every field the dictionary expects, correctly typed, before it is integrated.

For each candidate year, produce a schema check table:

| Field | Present | Type OK | Value | Flag |
|-------|---------|---------|-------|------|

Flag any field that is missing, wrong type, or has a value that violates a framework rule (see Part 3).

---

## PART 3 — FRAMEWORK RULE VERIFICATION

Apply each rule to every candidate year. Flag violations.

### 3A — Restructuring Two-Line Rule
The app stores a single `restructuring` value. It must equal the sum of BOTH income-statement restructuring lines (the AVGO pattern) — not just one. Verify the candidate's restructuring captures both lines summed. If the source 10-K shows two restructuring lines and only one appears reflected, flag it. A null where the source clearly discloses a charge is a flag.

### 3B — SBC MAX Rule
`Real SBC = MAX(SBC expense, SBC grant-date fair value)` — the app's `sbc_expense` and `sbc_grant` fields (dictionary names). The audit verifies both values are present and properly sourced, not which is larger — that's applied downstream. Both must have values for an annual 10-K extraction.

### 3C — Multi-Year SBC Grant Division
If the source 10-K describes a multi-year equity grant (e.g., AVGO's two-year FY2025 award), the SBC grant-date fair-value field must reflect the **annualized** value (total grant divided by years), not the gross multi-year figure. If the candidate value looks like a gross multi-year award, flag it and request confirmation of the annualization.

### 3D — TTM Null-Fallback Rule
For TTM periods only: if the SBC grant-date fair-value field is null in the candidate, the rule is to fall back to the prior 10-K's adjusted (annualized) grant value as a floor. The audit applies this fallback automatically and notes the substitution. Annual 10-K extractions should not be null — flag if they are.

### 3E — Depreciation vs. D&A Coherence
The app stores total D&A (`da`) and depreciation-only (`depr_only`); there is no separate amortization field. Verify `depr_only` ≤ `da`, and that the implied amortization (`da − depr_only`) is reasonable for the company. If the 10-K discloses only total D&A without a depreciation split, leave `depr_only` null and flag rather than estimating. A `depr_only` greater than `da` is a flag.

### 3F — Buyback Field Coherence
`share_repurchases` (dollar amount) and `buyback_shares` (share count) should both be present or both absent for a given year. A mismatch is a flag. If shares were repurchased but the share count isn't disclosed cleanly in the 10-K, note the source page and leave `buyback_shares` as null with a flag.

### 3G — Period Anchor Price Sanity
The app stores one period anchor per pill: `end_of_period_price`. For an annual 10-K period this should be the fiscal-year-end close. The top-level current price (saved via "Save current") is used only for the latest pill, never written into an older pill's anchor. Flag any case where an older pill's `end_of_period_price` appears contaminated by the current price rather than holding the period-end close (the historical-price contamination pattern).

## PART 4 — BRIEF RECONCILIATION

Cross-check the candidate years' headline figures against what the REPA Brief reports for the same period(s). The Brief is what the REPA app rendered AFTER the candidate was loaded, so it is the user-visible truth — the master must match it.

For each candidate year, verify:

- Revenue, CFO, GAAP Net Income — direct match
- D&A total — direct match
- SBC (the value the app used after MAX) — verify the underlying inputs are correct
- Maintenance CapEx (D&A × 1.05) — recompute and compare
- Restructuring total — direct match
- REP (Full Cost) — recompute and compare
- Enterprise Value — recompute and compare
- The three-tier yield cascade (GAAP EY, REP Yield, Real Ownership Yield) — recompute and compare

Any divergence > 1% on any line item is a hard flag and must be resolved before the master is updated.

---

## PART 5 — DISCREPANCY TABLE

Organize all flags from Parts 2–4 in a single table, sorted by severity:

| Year | Field / Rule | Severity | Master Value | Candidate Value | Recommended Action |
|------|--------------|----------|--------------|-----------------|--------------------|

Severity levels:
- **HARD** — must be resolved before master is updated (rule violations, Brief mismatches > 1%, missing required fields)
- **SOFT** — should be addressed but does not block integration (annotation gaps, source page missing, minor type coercions)
- **INFO** — purely informational (e.g., "TTM null-fallback applied automatically — substituted FY2024 grant value of $X.XB")

If there are zero HARD flags, proceed to Part 6 with a single line: "No hard flags. Ready to integrate."

If there are HARD flags, stop and surface them. Do NOT produce the updated master JSON. Wait for user resolution.

---

## PART 6 — INTEGRATION

Once HARD flags are zero (or resolved), integrate the candidate years into the master:

- **Additive case:** new years that don't exist in the master for this ticker → append to the company's year array, sorted chronologically
- **Replacement case:** years that already exist in the master → replace the entire year object with the candidate, after confirming with the user that replacement (not merge) is intended
- **Schema migration case:** if the master entry is missing the buyback/dividend/price fields (or any other current field per the Field Dictionary) for older years, the audit may fill those fields if the candidate or a referenced source provides them. Otherwise leave as null and flag.

State explicitly which years are being added vs. replaced vs. left untouched.

---

## PART 7 — OUTPUT

**Backup confirmation prompt** (one line):
> "Confirm master portfolio JSON has been backed up. Reply 'backed up' to receive the updated master, or 'hold' to pause."

After confirmation, output the updated master portfolio JSON in its own code block.

**Output discipline (per standing instruction):**
- If corrections were applied to EXISTING data (replacement case or schema migration), use the "say ready" pattern: output the updated master JSON first, then ask the user to say "ready" before any follow-up message.
- If the integration was purely additive (new years into clean existing entries), output the updated master JSON normally without the "ready" pattern.

After the JSON, produce a short integration summary:

```
INTEGRATION SUMMARY — [TICKER]
Years added: [list]
Years replaced: [list]
Years left untouched: [count]
Schema migrations applied: [yes/no, with detail]
Soft/info flags carried forward: [count, with one-line description of each]
Master version after update: [date]
```

That is the entire deliverable. No PDFs. No Gemini prompt. No Report 3. No Annotated companion.
